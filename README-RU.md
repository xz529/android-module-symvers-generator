# android-module-symvers-generator

Утилита для реконструкции файла `Module.symvers` напрямую из скомпилированных файлов `.ko` (kernel object). Инструмент специально разработан для сборки модулей и портирования ядер Android, особенно при работе с современными структурами разделов (A/B архитектура, `vendor_boot`).

## Зачем это нужно?

`Module.symvers` — это артефакт сборки, генерируемый системой сборки ядра (при выполнении `make modules`), который в норме НЕ входит в прошивку устройства и остается только в дереве исходников разработчика. 

Когда производители выпускают неполные исходные коды ядра или не предоставляют этот файл, компиляция внешних модулей (out-of-tree, например, драйверов Wi-Fi или кастомных DSP) становится невозможной из-за ошибки `disagrees about version of symbol`.

Этот скрипт восстанавливает полноценный эквивалент `Module.symvers` из уже скомпилированных `.ko` файлов. Он использует секцию `__versions` (`CONFIG_MODVERSIONS`), которая содержит CRC-хэши каждого символа, реально используемого модулем. Это в точности те же самые данные, которые попали бы в стандартный `Module.symvers` при обычной сборке ядра.

## Поддерживаемые разделы

Инструмент умеет рекурсивно сканировать директории и извлекать данные из модулей ядра, разбросанных по современным разделам Android. Вы можете направить его прямо на распакованные дампы прошивки, содержащие:
* `vendor_ramdisk`
* `vendor_dlkm`
* `system_dlkm`
* `vendor`
* ...и любые другие кастомные разделы производителя с файлами `.ko`.

## Зависимости и файлы

* **`extract_symvers.py`**: Главный и единственный скрипт.
* **Без тяжелых зависимостей**: Требуются только утилиты `readelf` и `nm` из стандартного пакета `binutils`.
* **Безопасно для кросс-компиляции**: Скрипт принципиально НЕ использует `objcopy`. Стандартный `objcopy` часто не распознает ARM64 в кросс-окружениях без multi-target binutils, что является классической головной болью при реверс-инжиниринге Android.

## Использование

### 1. Обработка конкретных файлов `.ko`:
```bash
python3 extract_symvers.py path/to/mod1.ko path/to/mod2.ko > Module.symvers
```
### 2. Рекурсивное сканирование дампа прошивки (Рекомендуется):
Укажите скрипту корневую директорию, куда вы распаковали разделы Android:

```bash
python3 extract_symvers.py --scan /path/to/firmware_extracted > Module.symvers
```
Пример: Сканирование распакованного дерева ядра Android (например, TECNO), содержащего сотни .ko файлов, успешно генерирует файл Module.symvers с тысячами валидных записей.

### Как это работает технически

Каждая запись в секции __versions занимает ровно 64 байта:

``` c
struct modversion_info {
    unsigned long crc;   // 8 байт на arm64 (!), значащими являются младшие 4 байта
    char name[56];       // Имя символа
};
```
Чтобы избежать падений objcopy на ARM64, скрипт безопасно читает секцию с помощью readelf -x __versions <file>. Он парсит 64-байтные записи, и для каждого использованного символа ищет по всем предоставленным .ko файлам тот модуль, который реально его экспортирует (используя таблицу глобальных символов через nm).

### Ограничения (честно)
Не различает EXPORT_SYMBOL и EXPORT_SYMBOL_GPL: Скрипт записывает EXPORT_SYMBOL для всех строк. Это влияет только на проверку лицензионной совместимости во время сборки (MODULE_LICENSE), не на само совпадение CRC-хэшей. Ваш модуль скомпилируется и загрузится без проблем.
Пропускает неиспользуемые символы: Символы, которые экспортируются, но НЕ используются ни одним другим .ko файлом в этой же прошивке, не попадут в итоговый файл. Чтобы извлечь их CRC, потребовался бы сложный парсинг __kcrctab по смещениям, что не реализовано (и редко требуется, так как обычно это касается только тестовых модулей).
Коллизии имен: Модуль-экспортер определяется путем совпадения имени глобального символа (nm, тип T/D/R/B). В крайне маловероятном случае совпадения имен между разными модулями может быть выбран не тот модуль.
