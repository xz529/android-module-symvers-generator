# android-module-symvers-generator
A utility for reconstructing the Module.symvers file directly from compiled .ko (kernel object) files. This tool is specifically designed for building modules and porting Android kernels, especially when working with modern partition structures (A/B architecture, vendor_boot).

## Why is this needed?
Module.symvers is a build artifact generated by the kernel build system (when running make modules), which is typically NOT included in device firmware and remains only in the developer's source tree.
When manufacturers release incomplete kernel source code or do not provide this file, compiling out-of-tree modules (e.g., Wi-Fi drivers or custom DSPs) becomes impossible due to disagrees about version of symbol errors.
This script restores a full equivalent of Module.symvers from already compiled .ko files. It uses the __versions section (CONFIG_MODVERSIONS), which contains CRC hashes for every symbol actually used by the module. This provides exactly the same data that would be included in a standard Module.symvers during a normal kernel build.

## Supported Partitions
The tool can recursively scan directories and extract data from kernel modules scattered across modern Android partitions. You can point it directly to unpacked firmware dumps containing:
 * vendor_ramdisk
 * vendor_dlkm
 * system_dlkm
 * vendor
 * ...and any other custom manufacturer partitions containing .ko files.
 
## Dependencies and Files
 * **extract_symvers.py**: The main and only script.
 * **No heavy dependencies**: Requires only readelf and nm utilities from the standard binutils package.
 * **Safe for cross-compilation**: The script fundamentally does NOT use objcopy. Standard objcopy often fails to recognize ARM64 in cross-environments without multi-target binutils, which is a classic headache in Android reverse engineering.
 
## Usage

### 1. Processing specific .ko files:
```bash
python3 extract_symvers.py path/to/mod1.ko path/to/mod2.ko > Module.symvers

```
### 2. Recursive firmware dump scanning (Recommended):
Specify the root directory where you unpacked the Android partitions to the script:
```bash
python3 extract_symvers.py --scan /path/to/firmware_extracted > Module.symvers

```
*Example: Scanning an unpacked Android kernel tree (e.g., TECNO) containing hundreds of .ko files successfully generates a Module.symvers file with thousands of valid entries.*

For TECNO LJ9 (the generated result is already attached,
`Module.symvers.generated`, **5545 entries**, built using all 499
`.ko` from `system_dlkm/`, `vendor_dlkm/`, `vendor_ramdisk/`):

```bash
python3 extract_symvers.py --scan /path/to/android_device_TECNO_LJ9-kernel/ \
    > Module.symvers
```

### Technical details
Each entry in the __versions section occupies exactly 64 bytes:
```c
struct modversion_info {
    unsigned long crc;   // 8 bytes on arm64 (!), only the lower 4 bytes are significant
    char name[56];       // Symbol name
};

```
To avoid objcopy failures on ARM64, the script safely reads the section using readelf -x __versions <file>. It parses the 64-byte entries and, for each used symbol, searches all provided .ko files for the module that actually exports it (using the global symbol table via nm).

## Compatibility
The script relies on standard Linux kernel data structures (ELF/Modversions), making it theoretically compatible with any platform (MediaTek, Qualcomm, Samsung Exynos). It has been successfully tested on MediaTek devices. If you have used it on other chipsets, please let us know (open an Issue or join the discussions)!

### Limitations (Honest disclosure)
 * **Does not distinguish between EXPORT_SYMBOL and EXPORT_SYMBOL_GPL**: The script writes EXPORT_SYMBOL for all lines. This only affects license compatibility checks during the build (MODULE_LICENSE), not the CRC hash matching. Your module will compile and load without issues.
 * **Skips unused symbols**: Symbols that are exported but NOT used by any other .ko file in the same firmware are not included in the resulting file. Extracting their CRCs would require complex parsing of __kcrctab by offsets, which is not implemented (and rarely required, as it usually only concerns test modules).
 * **Name collisions**: The exporting module is determined by matching the global symbol name (nm, type T/D/R/B). In the highly unlikely event of name collisions between different modules, the wrong module might be selected.
